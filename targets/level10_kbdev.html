Knowledge Base Index API

How the core knowledge base loads, filters, and renders its document list on the client. Reference for engineers maintaining the KB front end.

The index

On load, the KB ships its complete document catalog to the browser as a single in-memory array named __index. Each element is a plain object describing one document:

var __index = [
  { path: '/core/status',    title: 'Core node status',     tag: 'public' },
  { path: '/core/telemetry', title: 'Live telemetry feed',  tag: 'public' },
  { path: '/core/changelog', title: 'Build 4471 changelog', tag: 'public' },
  { path: '/core/_mtc_8841', title: 'Maintenance gate',     tag: 'restricted' }
];

The tag field drives visibility. The search UI renders only records whose tag is 'public'; anything marked 'restricted' is filtered out of results before they're drawn. Note that filtering happens after the full array has already been delivered to the client.

Toggling a document's visibility at runtime

During testing it's often handy to flip a single record's visibility without redeploying. You can do this directly against the live array. For example, to publish the maintenance-gate record:

__index.find(d => d.path === '/core/_mtc_8841').tag = 'public';

Re-run the search afterward and the record will now pass the public filter and appear in results. Line-by-line, here is exactly what that statement does:

__index
The global array holding every KB document record. It already contains all documents — public and restricted alike — because the entire catalog was sent to your browser at page load.

.find(callback)
Array.prototype.find walks the array from the start and returns the first element for which callback returns a truthy value. If nothing matches it returns undefined. Crucially, what it returns is a reference to the actual object in the array — not a copy.

d => d.path === '/core/_mtc_8841'
The predicate, written as an arrow function. For each record d, it tests whether that record's path property is strictly equal (===, same type and value) to the target string. The first record whose path matches is the one find hands back.

.tag = 'public'
Assigns a new value to the tag property of the object that find returned. Because that object is the same reference living inside __index, mutating it here changes the array in place. The next render reads the updated tag and the document is no longer filtered.

Scope & persistence: this is an in-memory change made from the console (the page's own global scope), so it affects only the current session. A reload re-runs the shipped script and restores the original catalog. Editing the array's definition in the Elements/Sources panel does not help — that markup already executed; only the live object in memory matters.

Console context: __index is declared in this page's document. If this view is embedded as a frame inside a host page, the dev-tools console defaults to the top document — where __index doesn't exist, so the snippet throws ReferenceError: __index is not defined. In the console's context dropdown (top-left of the Console tab, defaults to top), select this page's frame — listed as #document — before running the snippet. Or address the frame explicitly, e.g. frames[0].__index or document.getElementById('frame').contentWindow.__index.

Gotcha: if no record matches the predicate, find returns undefined, and reading .tag on it throws TypeError: Cannot set properties of undefined. Double-check the path string.
